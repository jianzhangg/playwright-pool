# init 体验优化 Design

**背景**

当前 `npx @jianzhangg/playwright-pool init` 对技术用户还能凑合，但对非技术用户存在 3 个明显问题：

1. 向导用词偏工程化，`profile`、`channel`、`slot` 这类词缺少业务解释。
2. 初始化确认后，如果源浏览器目录仍被占用，会在真正复制之前报错，但配置文件和部分目录已经落盘，让用户误以为“失败了却又初始化了一半”。
3. profile 复制可能持续较长时间，终端几乎没有进度信息，用户不知道当前是否仍在工作，也不知道能不能安全关闭窗口。

用户截图里的失败就是典型例子：

- 向导已确认
- `initializePlaywrightPool()` 先写配置、建目录
- `prepareProfiles()` 里 `ensureProfileDirClosed()` 才发现浏览器仍占用
- 最终报错退出，但运行目录里已经出现部分文件

**目标**

1. 对非技术用户，把向导和初始化过程改成“更容易看懂”的文案。
2. 在真正落盘前先做关键预检查，避免“浏览器没关导致失败但目录已半成品”。
3. 复制过程中持续输出明确的阶段状态，让用户知道程序还活着、当前在哪一步。
4. 在初始化失败时尽量清理当前这次新建的半成品，并给出明确的下一步提示。

**备选方案**

### 方案 A：只改文案

- 保留现有初始化顺序
- 仅把 `profile`、`channel`、`slot` 这些词换成更友好的中文

优点：

- 改动最小

缺点：

- 不能解决“失败后留下半成品”的核心问题
- 复制过程仍然没有状态反馈

### 方案 B：预检查 + 进度提示 + 失败清理

- 把“浏览器是否完全关闭”的检查提前到写配置前
- 初始化过程拆成清晰的阶段提示
- 如果这次运行新建了运行目录或配置文件，失败时尽量清理

优点：

- 同时解决截图里的核心体验问题
- 改动仍集中在 `init` 链路，风险可控

缺点：

- 需要调整 `initializePlaywrightPool()` 与 `prepareProfiles()` 的接口

### 方案 C：完整事务化初始化

- 所有产物先写到临时目录
- 最后一次性切换到正式目录

优点：

- 理论上最干净

缺点：

- 改动明显更大
- 需要处理已有运行目录、已有配置文件、已有 profile 目录的迁移与替换语义

**推荐方案**

采用方案 B。

原因：

- 方案 B 已经足够解决当前真实痛点
- 不需要引入完整事务化目录切换那种高复杂度改造
- 能在短期内发布一个对首次用户明显更友好的版本

**设计**

### 1. 向导文案改为用户语言

重点改这几类文本：

- `profile` -> `浏览器数据目录`
- `slot 数量` -> `浏览器副本数量`
- `确认写入配置并准备 profile` -> `确认开始初始化`
- 汇总信息里不再强调 `channel`

同时补一句明确提示：

- 初始化开始后请保持终端打开
- 复制浏览器数据可能需要几分钟

### 2. 初始化改成“预检查优先”

在 `initializePlaywrightPool()` 中把以下步骤提前到任何落盘前：

- 校验 `sourceProfileDir`
- 检查源浏览器目录是否仍被浏览器进程占用

这样浏览器没关时：

- 不会先写 `config.toml`
- 不会先创建 `logs/leases`
- 不会留下“刚开始就失败却已有半成品”的体验

### 3. 初始化进度提示

引入简单的进度回调，而不是复杂 spinner：

- 第 1 步：检查浏览器是否已完全关闭
- 第 2 步：创建运行目录并写入配置
- 第 3 步：准备浏览器副本
- 第 3 步内部：显示当前复制到第几个副本，例如 `正在准备浏览器副本 2/10`
- 第 4 步：初始化完成

这些提示全部使用中文短句，适合终端直接阅读。

### 4. 失败清理

如果本次运行是首次初始化，且 `initializePlaywrightPool()` 在写入配置后失败：

- 若运行目录是这次新建的，清理整个运行目录
- 若运行目录原本已存在，但配置文件是这次新建的，至少删除新写入的 `config.toml`

同时输出友好提示：

- 如果已经自动清理，明确告诉用户“未完成初始化，已清理本次新建文件”
- 如果无法完全清理，也明确告诉用户“可能保留部分中间文件，可删除运行目录后重试”

### 5. 用户可读错误

对这类典型错误做更友好的 CLI 输出：

- 浏览器仍占用源目录
- 源目录不存在
- 复制失败

原则：

- 对用户展示简短中文说明和下一步动作
- 不直接把长栈打到终端
- 保留内部错误链路供测试和开发定位

**不处理范围**

本次不处理：

- init 完成后自动帮用户关闭浏览器
- GUI 向导
- 复杂进度条 / 百分比估算
- 完整事务化 staging 目录切换

**测试策略**

1. `tests/init.test.ts`
   - 浏览器仍占用时不会写配置，也不会创建运行目录
   - 写入后复制失败时，会执行对应清理逻辑
   - 初始化阶段会触发预期的进度消息

2. `tests/init-wizard.test.ts`
   - 文案改成更易懂的中文
   - 仍保留取消、默认值、手动输入路径等现有行为

3. `tests/server-init.test.ts`
   - `runInitCommand()` 会把进度输出接到交互 IO
   - 用户取消时不会进入初始化

**风险与约束**

- 自动清理必须谨慎，不能误删用户已有运行目录
- 文案改动要避免过度冗长，否则交互会更啰嗦
- 进度提示要稳定，不能因为每个小文件都打印而刷屏
